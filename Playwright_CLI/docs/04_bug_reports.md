## BUG-CLI-01 — Password field has no visibility toggle

| Field | Detail |
|---|---|
| **Bug ID** | BUG-CLI-01 |
| **JIRA Ticket** | KAN-1 (logged in JIRA — somasaicheviti.atlassian.net) |
| **Reported by** | Soma Sai Dinesh Cheviti |
| **Date** | April 2026 |
| **Environment** | Chrome 124, app.vwo.com/#/login |
| **Severity** | Medium |
| **Priority** | Low |
| **Status** | Open |

### Description
The password input field on the VWO login page does not have a show/hide toggle button. Users must type the password blindly with no ability to verify what they have entered. This is a usability defect that increases login failure rates due to typos.

### Steps to Reproduce
1. Navigate to `https://app.vwo.com/#/login`
2. Click on the Password field
3. Type any password
4. Observe — no eye icon or toggle button to reveal the password

### Expected Result
A visibility toggle button (eye icon) should be present next to the password field. Clicking it should switch the input type between `password` (masked) and `text` (visible).

### Actual Result
No visibility toggle exists. The field is always masked. Users cannot verify their password before submitting.

### Impact
- Users with complex passwords are more likely to mistype
- Increased failed login attempts and support requests
- Does not meet WCAG 2.1 accessibility recommendation for password fields

### Workaround
None available in the UI. Users must retype carefully.

## BUG-CLI-02 — navigate() using `/login` instead of `/#/login` — all UI tests fail

| Field | Detail |
|---|---|
| **Bug ID** | BUG-CLI-02 |
| **Type** | Automation bug — incorrect locator strategy |
| **Reported by** | Soma Sai Dinesh Cheviti |
| **Date** | April 2026 |
| **Severity** | High |
| **Priority** | Critical |
| **Status** | Fixed |

### Description
The `navigate()` method in `pages/LoginPage.ts` used `goto('/login')` instead of `goto('/#/login')`. VWO is a Single Page Application (SPA). The `/#/login` route is handled by the client-side JavaScript router — the `#` is mandatory. Without it, the browser sends a server request for `/login`, which returns 404. No login form ever loads, causing all 10 UI tests to fail with `element not found` on the email input.

### Steps to Reproduce
1. Run `npx playwright test --project=ui`
2. Observe all 10 tests fail immediately with `getByRole('textbox', { name: 'Email address' })` not found
3. Inspect the failure screenshot — shows a blank or 404 page, not the login form

### Root Cause
`pages/LoginPage.ts` line:
```typescript
await this.page.goto('/login');  // WRONG — server route
```

### Fix Applied
```typescript
await this.page.goto('/#/login');  // CORRECT — SPA client-side route
```

### Verification
All 10 UI tests passed after the fix: `10 passed (53.9s)`

### Learning
SPA routing uses hash-based (`#`) or history API routes. Always verify the exact URL format using the browser address bar or `npx playwright codegen` before writing `navigate()` methods.

## BUG-CLI-03 — loginButton using wrong button name `Continue` — fill times out

| Field | Detail |
|---|---|
| **Bug ID** | BUG-CLI-03 |
| **Type** | Automation bug — incorrect locator |
| **Reported by** | Soma Sai Dinesh Cheviti |
| **Date** | April 2026 |
| **Severity** | High |
| **Priority** | Critical |
| **Status** | Fixed |

### Description
The `loginButton` locator in `pages/LoginPage.ts` was set to `getByRole('button', { name: 'Continue', exact: true })`. The VWO login button is labelled `Sign in`, not `Continue`. Because the locator could not find the button, any test calling `login()` or `clickLogin()` would time out waiting for the fill action to complete.

### Steps to Reproduce
1. Run `npx playwright test --project=ui`
2. Observe tests timeout at `locator.fill` — waiting for email input but actually failing because the button name is wrong
3. Error: `Test timeout of 30000ms exceeded — locator.fill: Test timeout of 30000ms exceeded`

### Root Cause
`pages/LoginPage.ts` constructor:
```typescript
this.loginButton = page.getByRole('button', { name: 'Continue', exact: true }); // WRONG
```

Generated by Cowork without verifying against live VWO page.

### Fix Applied
```typescript
this.loginButton = page.getByRole('button', { name: 'Sign in', exact: true }); // CORRECT
```

Verified using `npx playwright codegen https://app.vwo.com/#/login`.

### Verification
All 10 UI tests passed after both BUG-CLI-02 and BUG-CLI-03 fixes: `10 passed (53.9s)`

### Learning
Always verify AI-generated or tool-generated locators against the live page using codegen before running tests. Never assume a tool-generated locator is correct without manual verification.
