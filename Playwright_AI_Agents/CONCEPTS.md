## Approach Summary

Tests were **generated by Claude Code AI agents** — a planner agent audited the live VWO DOM and a generator agent produced 5 Playwright spec files. No test code was hand-written. This approach demonstrates AI-assisted test creation as a force multiplier for QA teams.

**Repo structure:**
```
Playwright_AI_Agents/
  playwright.config.ts               ← Global config (timeouts, reporters, browser)
  tests/vwo-login/
    page-load-smoke.spec.ts          ← 6 smoke tests (page title, inputs, buttons)
    invalid-credentials.spec.ts      ← Wrong password + partial input tests
    sql-injection-email.spec.ts      ← Security edge case
    empty-form-submission.spec.ts    ← Empty fields edge case
    forgot-password-visibility.spec.ts ← Forgot password form visibility
```

## 1. AI Agent Loop — Planner + Generator Pattern

**What it is:** A two-agent pipeline where:
- **Planner agent** — given a goal, audits the live application DOM, identifies all interactive elements, and produces a structured test plan
- **Generator agent** — consumes the planner's output and writes executable Playwright test code

**Where used:** The entire `tests/vwo-login/` folder was generated by this pipeline. No file in this folder was manually written.

**Why used:** Writing 5 spec files manually would take 2-4 hours. The AI agent pipeline produced them in ~15 minutes. The agent also identified edge cases (SQL injection, whitespace, empty forms) that a developer focused on happy paths might omit.

**Limitations:**
- AI assumes application behaviour based on common patterns — it got the SQL injection error text wrong (TC-edge-01 fix required)
- AI does not observe the live running app — it writes tests from DOM structure, not from observed behaviour
- No single owner for maintenance when VWO's DOM changes

**Alternatives:** 
- Manual test writing (slower but more accurate)
- Codegen with `npx playwright codegen https://app.vwo.com/#/login` — records browser actions into Playwright code (generates locators automatically, but no assertions)

## 2. `test.describe()` — Test Grouping

**What it is:** A container that groups related tests under a named block. Tests inside share a label in the HTML report and can share `beforeEach`/`afterEach` hooks.

**Where in repo:**
```typescript
// page-load-smoke.spec.ts:10
test.describe('VWO Login — Page Load Smoke', () => {
  // all smoke tests live here
});
```

**Why used:** Groups the 6 smoke tests together in the HTML report under "VWO Login — Page Load Smoke". Without grouping, all tests appear as a flat list, making reports harder to read.

**Alternatives:** `test.describe.only()` — runs only this group (used for local debugging, never committed). `test.describe.parallel()` — runs tests inside this group in parallel threads.

---

## 3. `test.beforeEach()` — Setup Hook

**What it is:** A function that runs before every test in its `describe` block. Used to set up consistent preconditions (navigate to page, wait for load, etc.).

**Where in repo:**
```typescript
// page-load-smoke.spec.ts:16
test.beforeEach(async ({ page }) => {
  await page.goto('/#/login');
  await page.waitForLoadState('networkidle');
  await expect(
    page.getByRole('textbox', { name: 'Email address' })
  ).toBeVisible();
});
```

**Why used:** Every test in the smoke suite needs the login page loaded and the email input visible. Without `beforeEach`, each test would repeat 3 identical lines at the start. `beforeEach` also ensures tests are independent — if test 2 modifies the page, test 3 starts fresh (new page from `beforeEach`).

**Alternatives:**
- `beforeAll()` — runs once before all tests in the describe block (page is shared between tests — tests are NOT independent)
- `afterEach()` — cleanup after each test (rarely needed in Playwright since each test gets a fresh browser context by default)
- `test.use({...})` — set configuration for all tests in a describe block without a hook

## 4. `async/await` — Asynchronous Execution

**What it is:** JavaScript's model for handling operations that take time (network requests, DOM waits, file reads). `async` marks a function as asynchronous. `await` pauses execution until the Promise resolves.

**Where in repo:**
```typescript
// All test functions — e.g., page-load-smoke.spec.ts:22
test('TC-smoke-01: page title contains VWO', async ({ page }) => {
  //                                          ↑ async keyword
  test.skip(!!process.env.CI, '...');
  await expect(page).toHaveTitle(/VWO/);
  //↑ await — waits for Playwright to check the title (with retries)
});
```

**Why used:** Playwright's entire API is Promise-based. `await` makes asynchronous code read like synchronous code — top-to-bottom, no callback nesting. Without `await`, the next line would execute before the previous one finishes (e.g., assertion runs before the page has loaded).

**What happens without `await`:**
```typescript
// WRONG — assertion executes immediately, before title is set
expect(page).toHaveTitle(/VWO/);

// CORRECT — waits for the assertion to pass (with retries)
await expect(page).toHaveTitle(/VWO/);
```

**Alternatives:** `.then()` chaining (older JavaScript style — hard to read with multiple async operations). Generator functions with `yield` (not used in modern test code).

## 5. `page.goto()` with `baseURL`

**What it is:** Navigates the browser to a URL. When `baseURL` is configured in `playwright.config.ts`, you pass only the path — Playwright prefixes the base automatically.

**Where in repo:**
```typescript
// page-load-smoke.spec.ts:17
await page.goto('/#/login');
// Resolved to: https://app.vwo.com/#/login (baseURL from config)

// playwright.config.ts:22
use: {
  baseURL: 'https://app.vwo.com',
}
```

**Why used:** `baseURL` means every test file contains `'/#/login'` instead of `'https://app.vwo.com/#/login'`. If VWO changes their domain, one config change updates all tests simultaneously.

**`goto()` options:**
```typescript
await page.goto('/#/login', {
  waitUntil: 'networkidle', // wait for no network activity
  timeout: 30000,           // override for this navigation only
});
```

**`waitUntil` values:**
| Value | Waits until |
|---|---|
| `'load'` | `window.load` event fires (all resources loaded) |
| `'domcontentloaded'` | DOM is parsed (fast, but CSS/images may not be ready) |
| `'networkidle'` | No network requests for 500ms (full SPA initialization) |
| `'commit'` | HTTP response headers received (fastest — no content wait) |

**Why `'/#/login'` not `'/login'`:** VWO uses Angular's hash-based routing (`#`). The hash fragment (`#/login`) tells the Angular router which "page" to render — it's not a real URL path that the server handles.

**Alternatives:** `page.reload()` — refreshes the current page. `page.goBack()` — browser back button. `page.waitForURL('/pattern/')` — waits for URL to match a pattern after navigation.

## 6. `page.waitForLoadState('networkidle')` — Wait Strategy

**What it is:** Pauses test execution until there are no network requests for at least 500 milliseconds. Indicates that the SPA has finished its initial data fetch and rendered.

**Where in repo:**
```typescript
// page-load-smoke.spec.ts:15
await page.waitForLoadState('networkidle');

// sql-injection-email.spec.ts:34
await page.waitForLoadState('networkidle', { timeout: 30000 });
```

**Why used:** VWO is an Angular SPA. After `page.goto()`, the browser receives the HTML shell but Angular hasn't bootstrapped yet. `waitForLoadState('networkidle')` waits for Angular's initialization requests to complete. In `sql-injection-email.spec.ts`, it replaces asserting a specific error text — VWO's SQL injection response takes a different code path, so we wait for the network to settle instead.

**Difference from `waitForLoadState('load')`:**
- `'load'` fires when all page resources (CSS, images, scripts) have finished downloading
- `'networkidle'` fires when the application has finished its own data requests after loading

For a SPA, `'networkidle'` is more meaningful — the page is "ready" when Angular's API calls have completed.

**When NOT to use `networkidle`:** Long-polling or WebSocket applications never reach "networkidle" because they continuously send network requests. Use `'load'` or explicit locator waits instead.

**Alternatives:**
```typescript
await expect(locator).toBeVisible();         // Wait for specific element (preferred)
await page.waitForSelector('.spinner', { state: 'hidden' }); // Wait for loading indicator to disappear
await page.waitForTimeout(3000);             // Fixed wait — anti-pattern (fragile)
```

## 7. `getByRole()` — ARIA-Based Locator

**What it is:** Locates elements by their ARIA role and optionally their accessible name. Reflects how screen readers and assistive technology navigate the page.

**Where in repo:**
```typescript
// page-load-smoke.spec.ts:16
page.getByRole('textbox', { name: 'Email address' })

// page-load-smoke.spec.ts:45
page.getByRole('button', { name: 'Sign in', exact: true })

// sql-injection-email.spec.ts:27
page.getByRole('textbox', { name: 'Email address' }).fill("' OR '1'='1")
```

**Common ARIA roles:**
| Role | HTML elements matched |
|---|---|
| `'textbox'` | `<input type="text">`, `<input type="email">`, `<textarea>` |
| `'button'` | `<button>`, `<input type="button">`, elements with `role="button"` |
| `'img'` | `<img>`, elements with `role="img"` |
| `'link'` | `<a href="...">` |
| `'checkbox'` | `<input type="checkbox">` |
| `'heading'` | `<h1>`–`<h6>` |

**Options:**
- `{ name: 'Email address' }` — accessible name (from `aria-label`, `aria-labelledby`, `<label>`, or placeholder)
- `{ exact: true }` — name must match exactly (not just contain the string)
- `{ level: 2 }` — for headings, matches `<h2>` specifically

**Why used:** `getByRole()` is Playwright's recommended locator. It's stable because it's tied to semantics, not DOM structure. If VWO changes the button's CSS class or position, `getByRole('button', {name:'Sign in'})` still finds it.

**Important limitation found in this project:** `getByRole('textbox', {name:'Email address'})` fails in the MCP project when both the login and forgot-password forms are in the DOM (strict mode violation). The MCP project uses `page.locator('#login-username')` instead.

**Alternatives:**
```typescript
page.getByLabel('Email address')        // by <label> text or aria-label
page.getByPlaceholder('Enter email ID') // by placeholder attribute
page.getByText('Sign in')               // by visible text content
page.locator('#login-username')         // by CSS ID selector
page.locator('input[type="email"]')     // by CSS attribute selector
```

## 8. `getByText()` — Text-Based Locator

**What it is:** Locates elements that contain specific visible text content. Uses partial matching by default.

**Where in repo:**
```typescript
// sql-injection-email.spec.ts:40 (in html content check — not getByText but similar concept)
expect(pageContent).not.toContain('500 Internal Server Error');
```

**Why used:** Used for locating error messages, labels, and UI text that doesn't have a stable ARIA role. `getByText('Your email, password, IP')` finds VWO's error message by its actual visible text.

**Partial vs exact matching:**
```typescript
page.getByText('Email')         // matches any element containing "Email"
page.getByText('Email', { exact: true }) // matches only exact text "Email"
```

**Alternatives:**
```typescript
page.getByRole('alert')               // for error messages with role="alert"
page.locator('[data-testid="error"]') // if the app uses test IDs
page.locator('.error-message')        // by CSS class (fragile)
```

## 9. `expect().toBeVisible()` — Visibility Assertion

**What it is:** Asserts that an element is present in the DOM AND visible (not hidden by CSS). Playwright retries this assertion until it passes or the timeout expires.

**Where in repo:**
```typescript
// page-load-smoke.spec.ts:38
await expect(
  page.getByRole('textbox', { name: 'Email address' })
).toBeVisible();

// sql-injection-email.spec.ts:47
await expect(
  page.getByRole('button', { name: 'Sign in', exact: true })
).toBeVisible({ timeout: 10000 });
```

**Why Playwright retries:** This is Playwright's "auto-waiting" — `expect().toBeVisible()` is NOT a one-shot check. It keeps re-evaluating the locator every 100ms until:
- The element becomes visible → assertion passes
- The timeout expires → assertion fails

This auto-waiting is why Playwright tests are more stable than Selenium tests, which require explicit `WebDriverWait` calls.

**Related assertions:**
```typescript
await expect(element).toBeHidden();         // element is not visible
await expect(element).toBeEnabled();        // button/input is not disabled
await expect(element).toBeDisabled();       // button/input is disabled
await expect(element).toBeChecked();        // checkbox is checked
await expect(element).not.toBeChecked();    // checkbox is not checked
await expect(element).toHaveValue('text');  // input has specific value
await expect(element).toHaveAttribute('type', 'password'); // has attribute
await expect(page).toHaveURL(/pattern/);    // page URL matches pattern
await expect(page).toHaveTitle(/VWO/);      // page title matches pattern
```

## 10. `test.skip(condition, reason)` — Conditional Skip

**What it is:** Skips a test at runtime when the condition is `true`. The test appears in the HTML report as "skipped" with the reason string visible.

**Where in repo:**
```typescript
// page-load-smoke.spec.ts:22
test.skip(
  !!process.env.CI,
  'VWO page title is set asynchronously by Angular SPA — inconsistent in headless CI. Verify locally.'
);

// page-load-smoke.spec.ts:29
test.skip(!!process.env.CI, 'Logo accessible name is inconsistent in headless CI');
```

**`!!process.env.CI` explained:**
- `process.env.CI` is the environment variable GitHub Actions sets automatically to `"true"` on every CI run
- `!!` converts the string `"true"` to boolean `true` (double negation)
- Result: condition is `true` in CI → test skips; condition is `false` locally → test runs

**Why used over removing the test:** The tests are valid for local verification. Removing them would delete the test intent. `test.skip()` preserves the test, makes the skip reason visible, and documents the environment inconsistency for future engineers.

**Variants:**
```typescript
test.skip();                              // unconditional — always skips
test.skip(condition);                     // skips when condition is true
test.skip(condition, 'reason string');    // skips with documented reason (recommended)
test.fixme(condition, 'reason string');   // similar but signals "needs fixing" (semantic difference)
```

## 11. `test.slow()` — Timeout Tripler

**What it is:** Triples the timeout for the test it's called in. If the global test timeout is 60s, calling `test.slow()` gives that test 180s.

**Where in repo:**
```typescript
// sql-injection-email.spec.ts:25
test.slow(); // VWO server validation takes 14-17s
```

**Why used:** The SQL injection test needs to wait up to 30s for `waitForLoadState('networkidle')` (VWO's server processes the SQL injection payload). A normal 60s test timeout could still be hit if VWO is slow. `test.slow()` gives 180s without changing the global config.

**Alternatives:**
```typescript
test('name', async ({ page }) => {
  test.setTimeout(120000); // set an explicit timeout for this test
});
```
`test.slow()` is cleaner than `test.setTimeout(bigNumber)` because it scales relative to the configured timeout rather than hardcoding a value.

## 12. `process.env.CI` — Environment Detection

**What it is:** A Node.js global object (`process.env`) that contains all environment variables. `process.env.CI` specifically is set to `"true"` by GitHub Actions (and most CI systems) automatically.

**Where in repo:**
```typescript
// page-load-smoke.spec.ts:22
test.skip(!!process.env.CI, '...');

// playwright.config.ts
forbidOnly: !!process.env.CI,
retries: process.env.CI ? 1 : 0,
workers: process.env.CI ? 1 : undefined,
```

**Why used:** Allows the same code to behave differently in local vs CI environments without separate config files. `process.env.CI ? 1 : 0` means "1 retry in CI, 0 retries locally."

**Pattern:**
```typescript
const isCI = !!process.env.CI;  // true in CI, false locally
```

**Other environment variables used in this portfolio:**
| Variable | Set by | Read in |
|---|---|---|
| `CI` | GitHub Actions automatically | All playwright.config.ts files |
| `VWO_TEST_EMAIL` | GitHub Secret | MCP project tests |
| `VWO_TEST_PASSWORD` | GitHub Secret | MCP project tests |
| `REQRES_API_KEY` | GitHub Secret | Playwright_CLI api tests |

## 13. `page.content()` — Raw HTML Access

**What it is:** Returns the full HTML source of the current page as a string. Unlike DOM queries, this gives you the raw text including all attributes, scripts, and metadata.

**Where in repo:**
```typescript
// sql-injection-email.spec.ts:39
const pageContent = await page.content();
expect(pageContent).not.toContain('500 Internal Server Error');
expect(pageContent).not.toContain('stack trace');
```

**Why used:** After a SQL injection attempt, we can't assert a specific error message (VWO's response varies). Instead, we scan the raw HTML to confirm no server error string appears. This is a security assertion — checking that the server didn't expose an HTTP 500 error page or a stack trace.

**Limitation:** `page.content()` returns the entire HTML (hundreds of kilobytes). Searching for `'500'` in this string is unreliable because the number 500 appears in many legitimate contexts (YouTube embed `width=500`, CSS values, JS config). The fix was to search for `'500 Internal Server Error'` — specific enough to only match a real HTTP error.

**Alternatives:**
```typescript
page.on('response', response => {
  if (response.status() === 500) { /* ... */ }
}); // Intercept HTTP responses — more precise
```

## 14. `playwright.config.ts` — Configuration File

**What it is:** The central configuration file for the entire test suite. Defines global settings like timeouts, reporters, browsers, and base URL.

**Where in repo:** `Playwright_AI_Agents/playwright.config.ts`

**Key settings and why:**
```typescript
export default defineConfig({
  testDir: './tests',          // where to find spec files
  fullyParallel: false,        // run tests sequentially (one at a time)
  forbidOnly: !!process.env.CI, // fail CI if test.only() is committed
  retries: process.env.CI ? 1 : 0, // retry once on failure in CI
  workers: 1,                  // single worker — prevents VWO rate-limit
  timeout: 60000,              // 60s per test (test.slow() triples to 180s)
  expect: { timeout: 15000 },  // 15s for any assertion to pass
  use: {
    baseURL: 'https://app.vwo.com', // prefix for page.goto('/#/login')
    actionTimeout: 20000,           // 20s max for any click/fill/etc.
    trace: 'on-first-retry',        // record trace when test retries
    screenshot: 'only-on-failure',  // screenshot when test fails
  },
  projects: [
    { name: 'chromium', use: { ...devices['Desktop Chrome'] } },
  ],
  reporter: [['html'], ['list']],
});
```

### `forbidOnly: !!process.env.CI`
**What it does:** If any test in the repo has `test.only()` committed, the entire CI run fails. Prevents engineers from accidentally committing `test.only()` which would only run one test in CI while all others are silently skipped.

### `retries: process.env.CI ? 1 : 0`
**What it does:** In CI, each failed test gets one retry before being marked red. Locally, tests fail immediately (no retry noise during development).

### `trace: 'on-first-retry'`
**What it does:** When a test fails and Playwright retries it, Playwright records a trace of the retry — every page action, DOM snapshot, and network request. The trace is uploaded as a CI artifact and viewable at `trace.playwright.dev`.

### `screenshot: 'only-on-failure'`
**What it does:** Takes a full-page screenshot only when a test fails. Stored in `test-results/` and uploaded as a CI artifact. Lets you see exactly what the page looked like when the test failed.

## 15. HTML Reporter

**What it is:** Playwright's built-in report that produces an interactive HTML page showing all test results — pass/fail status, test steps, screenshots, traces, and error messages.

**Where in repo:**
```typescript
// playwright.config.ts
reporter: [['html'], ['list']],
```

**Why dual reporter (`html` + `list`):**
- `'list'` — outputs test results line-by-line to the terminal as tests run (real-time feedback during CI run)
- `'html'` — generates a full report in `playwright-report/index.html` after all tests complete (browsable, filterable, shows screenshots and traces)

**Accessing the HTML report:**
```bash
# After CI run — download artifact from GitHub Actions
npx playwright show-report playwright-report/
```

**Alternatives:**
```typescript
reporter: [
  ['junit', { outputFile: 'results.xml' }], // for Jenkins/Azure DevOps integration
  ['dot'],     // minimal — one dot per test (good for very large suites)
  ['json', { outputFile: 'results.json' }], // machine-readable
  ['allure-playwright'], // Allure reporter (requires separate install)
]
```

## 16. Chromium-Only in CI

**What it is:** The CI step installs and runs only Chromium, not Firefox or WebKit.

**Where in repo:**
```yaml
# .github/workflows/playwright.yml
- name: Install Playwright browsers (chromium)
  run: npx playwright install --with-deps chromium

- name: Run tests
  run: npx playwright test
```

**Why only Chromium in CI:** Chromium covers ~70% of global browser usage. Installing all 3 browsers adds 3-4 minutes to CI and increases cost. Full cross-browser runs should be in dedicated nightly pipelines. The AI agents project runs `npx playwright test` without `--project` because it only defines one project (chromium) in the config.

**Alternatives:**
```bash
npx playwright install --with-deps  # install all browsers
npx playwright test --project firefox  # run specific project
```
